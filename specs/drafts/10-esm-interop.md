# Plan #10: ESM import-interop validation for the CJS package

## Problem
VISION (`specs/VISION.md` lines 45-46) explicitly declares the package is "Not ESM-first" — it ships as CommonJS and relies on Node's CJS-ESM interop for `import` consumers. However `package.json` (lines 8-14) declares only `types`/`require`/`default` conditions in `exports`, and there is no test under `test/` that exercises an ESM consumer (`import xifty from '@xifty/xifty'` or `import { extract } from '@xifty/xifty'`). README (lines 116-120) already shows an ESM `import` example to users, so the unverified-but-advertised behavior is a credibility risk: if Node's CJS interop ever regresses or a downstream tool resolves `exports` differently, no test catches it. The acceptance criteria also requires the packed-tarball smoke path to confirm an ESM consumer works end-to-end (parity with VISION principle 4).

## Approach
Add a single ESM-interop smoke test that spawns a `node` subprocess running a `.mjs` file which `import`s `@xifty/xifty` (both default and named imports) and asserts the surface matches the CJS surface. This mirrors the existing subprocess pattern already used in `test/binding.test.js` lines 101-124 (the `consumer installs do not switch to source-build mode` test uses `spawnSync(process.execPath, ...)`). The `.mjs` fixture lives under `test/fixtures/esm-consumer.mjs` and is spawned with `cwd` set to repo root so Node resolves `@xifty/xifty` via the repo's own `package.json` self-reference (Node 20+ supports package self-references when `name` is set, which it is). Additionally, extend `scripts/smoke-packed-package.js` (lines 76-87) so the packed tarball smoke also exercises an ESM `import` from the temp consumer dir — this is the parity gate that proves end users on ESM projects work. No change to `exports` is needed if the interop test passes; if it fails, the plan's fallback is to add an `import` condition pointing at `dist/index.js` (CJS interop in Node treats this as the default export of the namespace), which is exactly what Node's interop already does implicitly — so the additive change is documentation in README confirming ESM is supported and tested.

## Files to touch
- `test/binding.test.js` — add a new subtest that spawns the ESM consumer fixture and asserts exit 0 + expected stdout (mirrors the spawnSync pattern at lines 101-124).
- `scripts/smoke-packed-package.js` — after the existing CJS `requireFromTemp` block (lines 76-87), add an ESM consumer probe: write a `.mjs` file into `tempDir`, spawn `node` to run it, assert the same `packageVersion` and a `normalized` extract works.
- `README.md` — add a short "ESM" subsection (next to the existing TypeScript example at lines 110-120) confirming `import` is supported via Node's CJS interop and listing both default and named import forms.
- `specs/SRS.md` — add a one-line entry under section 4 (Validation Configuration) noting that the `test` and `verify-package` gates now cover ESM-interop.

## New files
- `test/fixtures/esm-consumer.mjs` — minimal ESM consumer: `import xifty, { extract, packageVersion } from '@xifty/xifty';` then run `packageVersion()`, `extract(fixturePath, { view: 'normalized' })`, print `OK <version>` on success or throw on mismatch. Receives the fixture path via `process.argv[2]` so the test owns the path resolution.

## Risks
- Package self-reference resolution: spawning the ESM fixture with `cwd` at repo root relies on Node's package self-reference (`@xifty/xifty` resolving to this repo via its own `package.json` `name`). This works in Node 20+ as long as `exports` is present (it is). Mitigation: if self-reference behaves oddly under `node --test`'s child-spawn env, fall back to spawning with `NODE_PATH` or using a `file:` URL import of `dist/index.js` via a small re-export shim — but the goal is to test the public package name, so prefer self-reference.
- The packed-tarball ESM probe runs inside a temp `npm init -y` consumer dir where `@xifty/xifty` is a real installed dep, so self-reference is not needed there — straight `import '@xifty/xifty'` works.
- Native binding load order: the ESM consumer transitively loads `dist/index.js` which calls `nodeGypBuild` at import-time. On a platform without a prebuild, the test will fail with the documented unsupported-platform error. This matches existing CJS test behavior (`test/binding.test.js` requires the binding to load), so no new platform exposure — but worth noting reviewers should not run this on unsupported platforms.
- `exports` field: the plan deliberately does NOT add an `import` condition. If a future Node version changes default-condition resolution, the test will catch it and a follow-up can add `"import": "./dist/index.js"` (which is what interop already resolves to). Per VISION non-goal "Not ESM-first," keeping `exports` minimal is correct.
- README drift: adding an ESM example that the test enforces creates a coupling — the test fixture's import shape must stay in sync with README. Mitigation: keep the fixture minimal and reference the same two import forms shown in README.
